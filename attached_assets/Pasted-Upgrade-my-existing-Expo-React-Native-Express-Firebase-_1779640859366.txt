====================================================
1. Balance system
====================================================

The app must use these balances:

energyBalance:
- Non-withdrawable.
- Earned from check-in, spin, scratch, Unity rewarded ads.
- Used for app benefits like unlocking extra task slots.
- Must never convert to PKR.
- Must never be withdrawable.

pendingCoinsBalance:
- Earned from offerwall/task providers only:
  monlix, tapjoy, ayet, pubscale.
- Not withdrawable.
- Status shown as Pending Verification.
- Can later become confirmed, rejected, or reversed.

confirmedCoinsBalance:
- Withdrawable.
- Converts to PKR using:
  1000 confirmed coins = PKR 20
- Withdrawals must only use confirmedCoinsBalance.

pkrBalance:
- Calculated from confirmedCoinsBalance only.

Do not allow mobile app to directly update sensitive balances.
All reward and withdrawal balance changes must happen on backend using Firestore transactions.

- createdAt

- reversalReason

All provider rewards must follow this flow:

Provider postback received
↓
Validate provider payload/signature if secret exists
↓
Check duplicate externalTransactionId + provider
↓
Create offerEvents record
↓
Credit pendingCoinsBalance only
↓
Create ledger/transaction entry
↓
Show user Pending Verification
↓
After hold period or admin approval
↓
Move from pendingCoinsBalance to confirmedCoinsBalance
↓
Only confirmedCoinsBalance can be withdrawn

====================================================
4. Reversal and chargeback handling
====================================================

If provider sends rejected/chargeback/reversal:

If reward is still pending:
- Deduct coinsCalculated from pendingCoinsBalance.
- Mark offerEvent status rejected or reversed.
- Mark transaction rejected/reversed.
- Save reason.
- Add fraud/risk note.

If reward was already confirmed:
- Deduct from confirmedCoinsBalance if possible.
- Update pkrBalance.
- Mark offerEvent status reversed.
- Mark transaction reversed.
- Increase suspiciousScore.
- Set manualReviewRequired true if needed.

Do not allow negative balances unless explicitly handled safely.
Use Firestore transactions.
